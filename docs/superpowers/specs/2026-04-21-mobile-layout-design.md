# 手机端布局优化设计

**目标**

优化 `whatToEat` 在手机端的使用体验：
- 主界面只保留转盘和最近记录
- 抽中结果通过短提示反馈，不再占用常驻结果卡片
- 配置界面调整为更符合手机端操作习惯的“列表主视图 + 底部抽屉”

**范围**

- 仅调整手机断点下的布局与交互
- 桌面端保留现有信息架构和主要交互方式
- 不引入新框架，不拆分现有单文件结构
- 不修改本地存储结构和菜单/历史数据格式

**现状问题**

- 手机端主界面同时显示转盘、结果卡、最近记录，主次信息竞争明显
- 配置界面沿用桌面端“双区布局”思路，在手机上显得拥挤
- 批量输入和单项维护在手机上没有形成清晰层级
- 主界面抽中结果依赖常驻结果区域，占用了较多垂直空间

**设计决策**

## 1. 手机主界面

- 在手机断点下隐藏“今日结果”卡片
- 主界面只保留：
  - 转盘卡片
  - 最近记录卡片
- 抽中结果时，在转盘下方显示一条短提示
- 短提示在短时间后自动淡出
- 抽中结果仍然写入最近记录，确保反馈可追溯

### 交互说明

- 用户点击转盘中心开始抽取
- 动画结束后：
  - 更新最近记录
  - 在转盘下方显示短提示，例如“已抽中：番茄鸡蛋面”
- 若用户连续抽取，短提示内容应被新结果覆盖并重新计时

## 2. 手机配置界面

- 手机端配置页调整为“列表主视图 + 底部抽屉”
- 主区默认只展示当前菜单列表
- 顶部工具栏简化为：
  - `返回转盘`
  - `管理菜单`
- 点击 `管理菜单` 后，打开底部抽屉

### 底部抽屉内容

- 默认展示两个操作入口：
  - `新增单项`
  - `批量导入`
- 点击入口后，在抽屉内部切换到对应表单，不跳转页面

### 列表区交互

- 每一项在手机端显示为单行卡片
- 每项提供：
  - `编辑`
  - `删除`
- 不加入拖拽排序，避免误触和增加复杂度

### 编辑方式

- 点击 `编辑` 后，当前项进入原地编辑状态
- 用户可直接修改内容
- 通过完成按钮或失焦保存

### 批量导入方式

- 批量导入表单位于底部抽屉中
- 提供多行输入框和 `加入列表` 按钮
- 导入成功后自动收起抽屉

**桌面端策略**

- 桌面端继续保留现有双区配置布局
- 桌面端保留“今日结果”卡片
- 手机端新增的底部抽屉和结果短提示为断点内专属体验

**实现边界**

- 仅修改：
  - `whatToEat/wheel.html`
  - `whatToEat/wheel.test.mjs`
- CSS 以现有断点体系为基础扩展
- JavaScript 在现有单文件逻辑中增加最小必要状态和视图切换逻辑

**验证要求**

- 为手机端布局新增回归测试，确保：
  - 手机端不显示“今日结果”卡片
  - 手机端主界面保留最近记录区域
  - 手机端存在底部抽屉相关结构或样式约束
  - 手机端抽中后存在短提示反馈逻辑
- 运行 `node wheel.test.mjs`
- 使用手机视口做一次实际页面检查，确认布局和可见性符合设计

**非目标**

- 不实现云端同步菜单
- 不增加账户体系或后端服务
- 不支持手机端拖拽排序
- 不对整体视觉风格做大规模重设计
