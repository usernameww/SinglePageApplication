# 配置页复制菜单设计

**目标**

在 `whatToEat` 的配置界面增加一个小型复制按钮，允许用户将“当前菜单”按“每行一项”的纯文本格式一键复制到系统剪贴板，复制结果可以直接粘贴进现有的“批量导入”输入框，无需额外转换。

**范围**

- 仅调整 `whatToEat` 配置界面的一个小交互
- 复制内容严格复用现有批量导入格式：每行一个菜单项
- 不修改本地存储结构，不引入新的数据模型
- 保持桌面端与手机端共用同一份配置页逻辑

**用户场景**

- 用户已维护好一份当前菜单，希望快速复制出来备用
- 用户想把当前菜单复制后直接粘贴回“批量导入”区域，用于合并、分享或跨设备中转
- 用户在菜单为空时不应得到无意义的空复制结果

**设计决策**

## 1. 按钮位置

- 在配置页“当前菜单”卡片标题行中增加一个小复制按钮
- 按钮靠近菜单列表标题，弱化视觉占用，但保持容易发现
- 不放到顶部工具栏，避免与页面级导航动作混在一起

## 2. 复制格式

- 将当前 `state.items` 中的 `label` 依次拼接为字符串
- 每个菜单项占一行，使用换行符连接
- 不添加编号、前缀、JSON 或额外说明文字
- 输出结果必须可以直接粘贴进现有的批量导入输入框并被现有导入逻辑正确解析

## 3. 状态反馈

- 复制成功后，通过现有状态文案区域给出轻量反馈
- 反馈内容应明确说明已复制多少项
- 菜单为空时，复制按钮禁用，避免触发空复制
- 如果浏览器不支持剪贴板写入或复制失败，状态区给出失败提示，不影响现有菜单内容

## 4. 实现边界

- 仅修改：
  - `whatToEat/wheel.html`
  - `whatToEat/wheel.test.mjs`
- 样式复用现有按钮体系，最多补一个小尺寸按钮样式
- JavaScript 只增加最小必要的文本导出和复制逻辑

**验证要求**

- 为复制按钮的存在与位置补充回归测试
- 为导出格式补充测试，确保输出是“每行一项”
- 为空菜单情况下的复制可用性补充测试或结构校验
- 运行 `node wheel.test.mjs`

**非目标**

- 不增加分享、下载文件、导出 JSON 等扩展能力
- 不增加复制成功 toast、弹窗或复杂动画
- 不改动批量导入的解析规则
